设计模式：单例模式

单例模式是Java开发中最为简单的设计模式之一，在实现方式上有许多种。所谓单例，就是这个类整个程序中有且只有一个实例，该类负责创建一个实例，同时也负责只有一个实例被创建。

特点

单例模式有以下几个特点：
1、构造方法私有。
2、持有自己类型的属性。
3、向外界提供一个静态方法，返回该类的唯一一个实例。

实现

饿汉模式

饿汉模式，顾名思义，就是在类加载的初始化阶段的时候，就进行实例化，好似个饿汉。

优点

该实现方法在类加载的时候就进行了实例化，在多线程的环境下不会产生线程安全问题。

缺点

由于是提前实例化，将会占用一定的资源。

懒汉模式

懒汉模式，顾名思义，就是等到需要的时候再进行实例化操作，像个懒汉。

优点

在需要的时候再进行实例化，节约了一定的资源。

缺点

当不加锁的时候有线程安全问题，加了锁会导致各线程串行请求获取实例，效率较低。

实现代码

双重锁模式

双重锁模式是对懒汉模式的进一步改进，代码里进行了两次实例是否存在的判断，第一次未加锁判断，是为了避免实例已初始化每次判断需要串行运行的情况，第二次加锁判断，是基于实例未初始化的情况下，考虑线程安全而采取的行动。

优点

弥补了懒汉模式下多线程串行运行导致效率低的弊端。

缺点

由于 singleton3 = new Singleton3();对象的创建在JVM中可能会进行重排序，导致返回一个未初始化完成的对象，在多线程访问下存在风险，使用volatile修饰signleton3实例变量能有效解决该问题。

实现代码

静态内部类模式

静态内部类模式是指实例的初始化交由静态内部类负责，只有当 getInstance()被调用的时候，虚拟机才开始加载静态内部类（静态内部类和非静态内部类一样，都是在被调用时才会被加载），并初始化实例，只有一个线程可以获得对象的初始化锁，其他线程无法进行初始化，保证对象的唯一性。

优点

这种实现方式是利用classloader的加载机制来实现懒加载，并保证构建单例的线程安全。

缺点

无法防止通过反射重复构建对象。

实现代码

枚举模式

枚举模式是根据枚举的特性而产生的,枚举类的每个项就是枚举类的每个实例。默认枚举实例的创建是线程安全的。

优点

使用枚举实现的单例模式，不仅可以防止利用反射强行构建单例对象，而且可以在枚举类对象被反序列化的时候，保证反序列的返回结果是同一对象。
对于其他方式实现的单例模式，如果既想要做到可序列化，又想要反序列化为同一对象，则必须实现readResolve方法。

缺点

单例在枚举类被加载的时候就初始化了，并非懒加载机制。
