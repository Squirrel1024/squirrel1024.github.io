HashMap和ConcurrentHashMap

HashMap 和 ConcurrentHashMap 都是 Java 中很常见的数据结构，因此就在这基于自己的认识对他们做一些小小的总结。

HashMap

HashMap 是一种存储键值对 Key-Value 的集合,这种键值对又叫Entry。HashMap 的底层数据结构是数组，Entry 则分散地存储在此数组中。

put()

调用put方法的时候，首先对put的key进行hash计算,具体做法就是将key的hashcode()与自己的前16高位做异或运算得出哈希值。之所以这么做，是因为HashMap的初始容量为16,与hashcode直接做与运算的话相当于只有hashcode的后四位有效。为了HashMap分布更加均匀，减少hash冲突，所以加入高位进行扰动。
其中还有一点需要注意，HashMap支持key为null的存储,null的哈希值为0。

得到key的哈希值后，进行寻址操作，寻址操作即是将得到的哈希值和HashMap的做与运算，从而定位到将要插入的位置。之所以不用常见的取模运算，是因为计算机对位运算的性能更高。

当产生哈希冲突时，HashMap采用的是链地址法，其中jdk1.7版本中HashMap采用的是头插法，因为编写者基于性能角度考虑，认为最后插入的结点访问的可能性更大，这样能更快地找到该结点，但这种写法在多线程环境下可能会产生环。在jdk1.8中，结点的插入改为尾插法，同时在链表长度达到阈值8的时候，会将链表优化成红黑树，提高查询效率。当找到相应插入位置的时候，进行插入结点，如果找到相同key的时候，则对此结点的value进行修改。以下是jdk1.8关于put()操作的相关源码。

get()

get()方法即是通过key在HashMap中找到相应的value。步骤与put方法相似，同样是先计算key的hash来定位，从而找到相应的结点，返回对应的value值。

容量的一些问题

HashMap的默认初始容量是16，建议设置容量为2的幂次方。原因是因为上文提到的HashMap的哈希计算是采用与运算。

如果采用的容量为10的话，10的二进制是1010,与哈希值做与运算，那么第一位和第三位的数字不管怎样都是0，大大增加哈希冲突的可能性，因此做以上建议。
除此之外，当容量达到规定的阈值之后，HashMap会进行扩容。这个阈值时根据HashMap长度*负载因子得到的。也就是说当HashMap.Size >= Capacity * LoadFactor，产生扩容动作。其中Capacity默认16，LoadFactor默认0.75f。
扩容动作包括以下两步：
1.创建一个新数组，长度为原来的两倍。
2.遍历原数组，根据新数组参数进行重新hash，并插入到新数组中。

线程安全问题

由于没有加锁，HashMap存在着线程安全问题，主要有以下两方面。
1.由于jdk1.7中采用了头插法，所以在多线程环境下，多个线程同时扩容时可能会使链表存在环。
2.当多个线程同时插入时，存在以下情况：A线程获取到头结点，A线程时间片结束；B线程获取到与A一样的头结点，并成功插入，B线程时间片结束。这时A线程时间片开始，将数据插入头结点后，此时将导致B结点的数据丢失。

ConcurrentHashMap

ConcurrentHashMap相比HashMap来说，具有线程安全的特性。在jdk1.7和jdk1.8有较大的不同，在此分开叙述。

ConcurrentHashMap1.7

ConcurrentHashMap在jdk1.7基于分段锁机制，采用的是segment数组+hashEntry数组+链表的数据结构来实现的。ConcurrentMap本质上是个segment数组，其中每个segment包含了若干的hashEntry,一个hashEntry下又连着链表，这样做的好处是降低了锁的粒度。


put()

当线程执行put()操作时，首先需要对key进行hash一次，定位segment的位置，然后再进行hash一次，定位hashEntry的位置，然后获取该segment的锁，从而执行插入动作。
segment的锁采用的是ReentrantLock,当插入的时候会调用tryLock（）方法尝试去获取锁，如果获取成功就直接插入相应的位置，如果已经有线程获取该Segment的锁，那当前线程会以自旋的方式去继续的调用tryLock（）方法去获取锁，超过指定次数就挂起，等待唤醒。

size()

当ConcurrentHashMap在执行size()的时候，可能会统计的同时，有线程执行put()操作

ConcurrentHashMap1.8

ConcurrentHashMap1.8摒弃了1.7采用的分段锁思想，采用的是Node数组+链表/红黑树的结构，其中，Node是ConcurrentHashMap存储结构的基本单元，继承于HashMap中的Entry，用于存储数据。同时通过CAS和synchornized来保证线程安全。虽然在JDK1.8中还能看到Segment的数据结构，但是已经简化了属性，只是为了兼容旧版本。可以说，ConcurrentHashMap1.8相当于线程安全版的HashMap。

put()

put()操作逻辑步骤如下：

  1. 如果没有初始化就先调用initTable（）方法来进行初始化过程
  2. 如果没有hash冲突就直接CAS插入
  3. 如果还在进行扩容操作就先进行扩容
  4. 如果存在hash冲突，就对该Node加锁来保证线程安全，这里有两种情况，一种是链表形式就直接遍历到尾端插入，一种是红黑树就按照红黑树结构插入，
  5. 最后一个如果该链表的数量大于阈值8，就要先转换成红黑树的结构，break再一次进入循环;如果添加成功就调用addCount（）方法统计size，并且检查是否需要扩容。

get()

get()方法的步骤与HashMap的相似，在此不再赘述。需要注意的是，get（）方法并没有通过加锁来保证自己获取的数据是最新的。ConcurrentHashMap是通过给Node加关键字volatile来保证可见性和有序性的。get操作可以无锁是由于Node的元素val和指针next是用volatile修饰的，在多线程环境下线程A修改结点的val或者新增节点的时候是对线程B可见的。

同时为了保证扩容的线程安全，ConcurrentHashMap 在底层存储数组加了volatile，保证扩容对其他线程可见。
